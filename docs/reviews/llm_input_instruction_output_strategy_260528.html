01 · Executive Summary

요약 결론

LLM은 문단을 새로 쓰는 생성기가 아니라 sentence_id 사이의 split 여부를 판정하는 teacher annotator로 써야 한다. 원천 파일의 차이는 모델 지시문보다 입력 포장 방식에서 먼저 흡수하고, 출력은 모든 문서에서 같은 JSON schema로 고정한다.

핵심 Takeaways

  • 입력: 원문 전체를 던지지 말고, extracted_blocks와 sentence_units를 만든 뒤 구조 metadata를 함께 보낸다.
  • instruction: 모든 문서 유형에 공통 system prompt를 쓰고, 원천 포맷별 주의사항만 user payload 안의 instructions에 추가한다.
  • 출력: boundary_annotations, proposed_atomic_paragraphs, quality_flags를 strict JSON으로 받는다.
이 문서가 답하는 질문
1. DATAlab HTML, DOCX, HWP를 각각 어떤 형태로 LLM에 넣을 것인가?
2. LLM에게 경계 판단을 시킬 때 system/user instruction은 어떻게 분리할 것인가?
3. classifier 학습과 human review에 바로 쓸 수 있는 출력은 무엇인가?

현재 smoke test에서 확인된 신호

case source_type effort sentences splits tokens 판정
datalab_xhigh datalab_parsed_json xhigh 8 4 5714 반복 heading과 HTML boundary hint를 구분함.
docx_high docx high 8 3 3375 한국어 본문만 남긴 뒤 scripture span을 안정적으로 보존함.
hwp_low hwp low 8 4 2327 짧은 scripture opening은 처리했지만 extraction noise flag가 필요함.
02 · Background

공통 I/O 계약

source_type이 달라도 LLM 앞단과 뒤단의 계약은 같아야 한다. 차이는 metadata richness, 구조 hint 신뢰도, extraction warning에만 반영한다.

LLM에 넣기 전 canonical flow

모든 원천 문서는 raw_document -> extracted_blocks -> normalized_blocks -> sentence_units -> teacher_inputs -> teacher_outputs -> atomic_paragraphs 순서로 통과한다. LLM은 이 흐름의 teacher_inputs 단계에만 들어간다.

공통 user payload

LLM에는 자연어 원고가 아니라 아래 형태의 JSON 문자열을 넣는다. 핵심은 sentence_id 안정성과 원천 metadata 보존이다.

{
  "task": "annotate_sentence_boundaries",
  "document_id": "stable_document_id",
  "source_type": "datalab_parsed_json | docx | hwp",
  "document_kind": "sermon | lecture | book_chapter | prayer | notice | thesis | unknown",
  "reasoning_effort": "low | high | xhigh",
  "source_path": "datas/...",
  "extraction_notes": ["parser/tool notes"],
  "allowed_boundary_types": [
    "none",
    "topic_shift",
    "scripture_reading_start",
    "scripture_explanation_start",
    "illustration_start",
    "application_start",
    "prayer_or_closing",
    "enumeration_start"
  ],
  "instructions": ["source-specific constraints"],
  "sentences": [
    {
      "sentence_id": "doc.s0001",
      "text": "문장 원문",
      "block_id": "b0001",
      "block_type": "heading | paragraph | quote | table | unknown",
      "source_tag": "h2 | p | docx_paragraph | hwp_paragraph",
      "page_id": null,
      "paragraph_index": 12,
      "heading_context": [],
      "html_boundary_before": false
    }
  ]
}

boundary taxonomy

boundary_type 쓸 때 금지할 때
none split_after=false인 모든 경우. split_after=true와 함께 쓰지 않는다.
topic_shift 제목, 인사, 주제 전환, 새 논지 시작. 질문과 바로 다음 답변 사이.
scripture_reading_start 본문 낭독 또는 짧은 재낭독이 시작될 때. 성경 구절 설명이 시작되는 문장.
scripture_explanation_start 인용이 끝나고 해석, 적용, 설명으로 넘어갈 때. 구절 번호만 분리하는 경우.
illustration_start 예화, 간증, 사례 서사가 시작될 때. 짧은 수사적 질문만 있는 경우.
application_start 청중에게 직접 적용하거나 권면이 시작될 때. 본문 해석의 자연스러운 연속.
prayer_or_closing 기도, 초청, 마무리 멘트. 설교 중간의 짧은 감탄문.
enumeration_start 첫째/둘째, 1장/2장, 항목형 설명이 시작될 때. 이미 같은 열거 안의 다음 항목일 뿐인 경우.
03 · Methodology

원천 포맷별 입력 전략

포맷별 차이는 원문 품질과 구조 정보의 신뢰도 차이다. 따라서 LLM에 넣는 텍스트 양보다 어떤 hint를 같이 넣는지가 더 중요하다.

1. datalab_parsed_json: HTML 구조를 약한 라벨로 사용

항목 전략
입력 단위 JSON 내부 html을 직접 보내지 말고, h1/h2/h3/p/li/table를 block으로 파싱한 뒤 sentence unit으로 보낸다.
필수 metadata source_tag, page_id, heading_context, html_boundary_before, 반복 heading 후보.
reasoning effort OCR 노이즈, 반복 제목, page header/footer가 섞이면 xhigh. 구조가 깨끗하면 high.
주의 HTML 경계는 강한 hint지만 정답은 아니다. LLM instruction에 "consider but do not blindly follow"를 반드시 넣는다.
받을 output 반복 heading, scripture-reference 결합, 원래 HTML paragraph 내부 semantic split 여부를 quality_flags로 받는다. 
{
  "source_type": "datalab_parsed_json",
  "extraction_notes": [
    "JSON html parsed with HTMLParser.",
    "h1/h2/p/page metadata retained as label hints.",
    "page_count=50"
  ],
  "instructions": [
    "HTML tag and heading_context are boundary hints, not gold labels.",
    "Detect repeated page headings and report them in quality_flags.",
    "Keep scripture text and verse reference in the same atomic paragraph where possible."
  ]
}

2. docx: 한국어 본문만 남기고 paragraph index를 보존

항목 전략
입력 단위 word/document.xml의 w:p를 paragraph 단위로 읽고, 그 안에서 sentence unit을 만든다.
필수 metadata paragraph_index, language_filter_reason, script_counts, 제거된 외국어 paragraph 수.
reasoning effort 한국어/외국어 번역 단락이 교대하면 high. 깨끗한 한국어 원고는 batch 비용을 보고 낮출 수 있다.
주의 제거된 외국어 본문은 LLM input에 다시 넣지 않는다. 대신 audit count와 paragraph id만 남긴다.
받을 output 제목, scripture epigraph, sermon orientation, definition/explanation unit을 atomic paragraph로 묶은 결과. 
{
  "source_type": "docx",
  "removed_foreign_paragraphs": 81,
  "extraction_notes": [
    "DOCX word/document.xml parsed by paragraph.",
    "Foreign translation paragraphs removed by script-ratio heuristic.",
    "raw_paragraphs=162"
  ],
  "instructions": [
    "Do not infer boundaries from removed foreign paragraphs.",
    "Treat rhetorical question and immediate answer as one semantic unit unless a new topic begins.",
    "Preserve verse quotation and verse reference as one scripture_reading paragraph."
  ]
}

3. hwp: extraction warning을 입력 계약에 포함

항목 전략
입력 단위 libhwp로 section과 paragraph를 순서대로 읽고 sentence unit을 만든다.
필수 metadata hwp_version, section id, paragraph_index, 추출 품질 warning, duplicate group 후보.
reasoning effort 짧은 smoke test는 low도 가능했지만, production label은 추출 warning이 있으면 high로 올린다.
주의 깨진 문자, 잘린 문장, 필사본/원본 중복을 LLM이 고치게 하지 않는다. 문제는 quality_flags로 표시하게 한다.
받을 output title/date line 분리, scripture reading cue, quotation, explanation cue, truncation/noise flag. 
{
  "source_type": "hwp",
  "extraction_notes": [
    "HWP parsed with libhwp.",
    "hwp_version=5.0.3.0",
    "paragraphs=61"
  ],
  "instructions": [
    "Do not repair possible extraction noise; flag it.",
    "If the excerpt ends mid-discourse, do not invent a following boundary.",
    "Separate scripture-reading setup, scripture quotation, and explanation cue when the transition is explicit."
  ]
}

담화 종류별 보정

source_type은 파일 포맷이고, document_kind는 내용 장르다. 같은 DOCX라도 설교, 공지, 기도문은 boundary 기준이 다르다.

document_kind 추가 입력 hint instruction 보정
sermon 본문 낭독, 해석, 예화, 적용, 기도 cue. 설교 담화 전환을 boundary_type으로 적극 분류.
lecture 장/절/번호, 질문-응답, 정의 문장. 열거와 정의 unit을 과분할하지 않음.
book_chapter heading hierarchy, page id, 반복 header 후보. heading은 강한 hint, page break는 약한 hint.
prayer 기도 시작/종결 표현, 아멘, 청원 전환. 짧은 청원문마다 split하지 않음.
notice 날짜, 장소, 대상, action item. 담화 boundary보다 항목 구조를 우선.
thesis 목차, 논문 heading, footnote/table 여부. 설교 특화 label보다 topic_shift, enumeration_start 중심.
04 · Conclusions

LLM instruction 작성법

system instruction은 역할과 금지를 짧게 고정하고, user instruction은 현재 batch의 포맷·품질·경계 판단 규칙을 담는다. 모델이 문단을 생성하게 만들지 말고, sentence id 위에서 판정하게 만든다.

권장 system instruction

You annotate Korean sermon discourse boundaries for RAG preprocessing.
Do not rewrite, summarize, translate, or normalize the sermon text.
Use only sentence_id references when proposing boundaries.
Return split_after for every provided sentence.
Prefer atomic semantic paragraphs, preserve scripture quotations with their references,
and mark boundary_type from the allowed taxonomy.
If metadata and semantics conflict, explain the conflict in rationale or quality_flags.

공통 user instruction

[
  "Return one boundary_annotations item for every provided sentence.",
  "Use boundary_type='none' when split_after=false.",
  "Keep Bible quotation spans together with verse references where possible.",
  "Treat headings, original paragraphs, and page boundaries as hints, not gold labels.",
  "Do not infer a split after the final sentence unless the boundary is explicit inside the excerpt.",
  "Report extraction noise, duplicate headings, truncation, and uncertain language filtering in quality_flags."
]

포맷별 instruction delta

source_type 추가 instruction 의도
datalab_parsed_json HTML tag, heading_context, page_id를 보되 반복 heading은 quality flag로 분리. 구조 정보를 활용하되 OCR/page 노이즈를 정답으로 착각하지 않게 함.
docx 제거된 외국어 paragraph를 고려하지 말고, 남은 한국어 sequence 안에서만 판단. 병렬 번역 구조가 설교 담화 구조로 학습되는 것을 막음.
hwp 추출 오류를 고치지 말고 flag로 남김. sample truncation이면 마지막 boundary를 보수적으로 처리. teacher label과 extraction QA를 분리함.
금지 prompt
"문단을 보기 좋게 다시 나눠줘", "자연스러운 문단으로 재작성해줘", "원문을 다듬어서 출력해줘" 같은 지시는 쓰지 않는다. 이 프로젝트의 목표는 예쁜 문단 생성이 아니라 classifier 학습 가능한 split_after 라벨 생성이다.
05 · Appendix

출력 JSON과 검수 규칙

필수 output shape

LLM 응답은 strict JSON schema로 받는다. downstream classifier에는 boundary_annotations가, RAG chunk builder에는 proposed_atomic_paragraphs가 들어간다.

{
  "document_id": "doc_id",
  "source_type": "docx",
  "reasoning_effort": "high",
  "preprocessing_observations": [
    "DOCX document.xml was parsed by paragraph."
  ],
  "boundary_annotations": [
    {
      "sentence_id": "doc.s0001",
      "text_excerpt": "요약 excerpt",
      "split_after": true,
      "boundary_type": "scripture_explanation_start",
      "confidence": 0.96,
      "rationale": "성경 인용이 끝나고 해석이 시작된다."
    }
  ],
  "proposed_atomic_paragraphs": [
    {
      "paragraph_id": "doc.p0001",
      "sentence_ids": ["doc.s0001", "doc.s0002"],
      "paragraph_role": "scripture_reading",
      "reason": "본문과 절 reference를 같은 span으로 보존."
    }
  ],
  "quality_flags": [
    "scripture_reference_kept_with_quote"
  ]
}

자동 검수 규칙

검수 항목 실패 조건 처리
coverage 입력 sentence 수와 boundary_annotations 수가 다름. 응답 폐기 후 같은 payload 재시도.
id integrity 없는 sentence_id를 참조하거나 id를 새로 만듦. schema 통과 여부와 별개로 hard fail.
boundary consistency split_after=false인데 boundary_type이 none이 아님. 자동 수정하지 말고 재요청.
paragraph coverage proposed_atomic_paragraphs가 sentence를 누락하거나 중복 포함. human review 또는 재요청.
scripture preservation 성경 본문과 바로 붙은 절 reference를 불필요하게 분리. 검수 큐로 보냄.
confidence routing confidence < 0.75, extraction warning, duplicate heading flag. human adjudication 우선 샘플.

학습 데이터 row로 변환

{
  "left_sentence_id": "doc.s0004",
  "right_sentence_id": "doc.s0005",
  "split_after_left": true,
  "boundary_type": "topic_shift",
  "teacher_confidence": 0.84,
  "source_type": "datalab_parsed_json",
  "document_kind": "book_chapter",
  "features": {
    "left_block_type": "paragraph",
    "right_block_type": "paragraph",
    "html_boundary_before_right": true,
    "same_original_paragraph": true,
    "heading_context_depth": 2
  },
  "review_status": "teacher_only | human_checked | adjudicated"
}

최종 운영 원칙

  1. LLM batch는 40-80 sentence window로 자르고, 앞뒤 2 sentence context를 추가하되 target id만 annotate하게 한다.
  2. 문서별 첫 batch에는 제목, 파일명에서 추출한 날짜/시리즈, document_kind를 넣어 담화 맥락을 준다.
  3. teacher label을 gold로 바로 쓰지 않고, 낮은 confidence와 quality_flags가 있는 샘플을 human adjudication으로 보낸다.
  4. atomic paragraph와 RAG chunk를 분리한다. LLM output은 atomic paragraph까지이고, retrieval chunk는 후처리 merge가 만든다.
